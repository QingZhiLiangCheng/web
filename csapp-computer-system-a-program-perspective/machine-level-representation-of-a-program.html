Gcc是gcc C编译器
-Og是一种编译器优化的级别，除Og之外还有O1，O2等更高级别的优化，但是使用更高级别的优化产生的代码会严重变形，导致产生的机器代码与最初的源码之间关系难以理解，所以我们采用Og的优化

Machine-Level Code

在前面 提到了operator system引入的抽象的三个东西 process，virtual address space，file
这些抽象的东西隐藏了实现的一些细节，比如说linux的哲学本质是：Everything is file(前面章节提到的)，比如说一个disk，程序员把disk抽象成一个文件，不需要考虑里面一些实现的细节，只需要对文件进行读写。
有两个对于 machine-level Programming的抽象比较重要：

  • 机器级程序的格式和行为是由instruction set architecture（ISA,指令集架构）定义
    ISA定义了处理器的状态，指令的格式以及每条指令对状态的影响
    x86-64 就是一个ISA
    他给我们一种假象？好像每个指令都是按照顺序执行的
    但事实上处理器硬件要复杂得多，它可以同时处理多条指令，其采用了安全措施来确保整体行为与ISA规定的顺序操作相匹配
  • machine-level programming 使用的内存地址是virtual address
    其实我们可以把每一个virtual address 可以看做一个大型的数组，对于virtual address我们前面其实提到过基本的结构

compiler进行的工作是：将C提供的相对抽象的执行模型表示的程序->处理器执行的非常基本的指令
其中汇编指令就非常接近机器代码，其中可以包含一些机器指令的特征，所以我们可以通过汇编指令来了解整个程序的运行过程

这里为什么程序一开始要保存寄存器rbx的内容？
需要简单介绍一下寄存器的背景知识
Intel x86-64的处理器中包含了16个通用目的的寄存器

图中所显示的这16个寄存器，都用来存储数据或者指针。他们的名字都是%r开头的。
在解释寄存器之前，我们要先搞清楚两个概念：

  • Caller-saved 调用者保存寄存器
  • Callee-saved Register 被调用者保存寄存器

    比如说图中的例子 函数A调用了函数B 看call指令
    所以说函数A是caller 函数B是callee
    由于调用了函数B 我们发现%rbx中的数据被修改了
    逻辑上寄存器rbx的内容在调用函数B的前后应该保持一致
    所以我们需要解决这个问题。解决这个问题有两种方法：
  • 函数A在调用函数B之前，提前保存寄存器rbx的内容 执行完函数后再恢复寄存器rbx原来存储的内容

    这个策略就叫Caller-Saved 调用者保存
  • 函数B在使用寄存器rbx之前，先保存寄存器rbx的值，在函数返回之前再恢复rbx中的内容

    这个策略叫做Callee-Saved 被调用者保存

对于使用哪一种策略，不同的寄存器有不同的策略

其中rbx被定义为被调用者保存寄存器，所以pushq就是用了保存rbx的内容 popq就是恢复寄存器rbx的内容

其中根据寄存器用法的定义，函数mulstore的三个参数分别保存在%rdi，%rsi，%rdx中
在movq %rdx %rbx执行完毕后，rbx和rdx中的内容一致，都是dest指针所指向的内存地址
mov指令的后缀q表示数据的大小

图中的表格给出了C语言的基本类型对应的汇编后缀表示
比如说move指令就有4个变种 分别是movb，movw，movl，movq
call指令对应的是C代码中的函数调用
该函数的返回值会保存在寄存器rax中，所以说rax寄存器中其实就是保存了x * y的值
下一条指令 movq %rax， （%rbx） 是指将rax中的值送入内存中，内存的地址就存放在寄存器rbx中

linux> gcc -Og -c mstore.c
